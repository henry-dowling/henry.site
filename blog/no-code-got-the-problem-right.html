No code got the problem right and the solution wrong

No code didn't quite work

TLDR:

  1. No-code nailed the problem and bet on the wrong solution
  2. There's an irreducible amount of complexity inherent in the development of any reasonably useful piece of software
  3. Apps = boilerplate + context

"No-code" was a very fashionable term in the early 2020s that referred to software that allowed other people to write software without needing to code.

The basic idea of "no-code" is to allow anyone (read: non-technical people) to write software by abstracting away the boilerplate parts of coding—setting up a db, hosting a server, building email integrations—until only the "business logic" part remains.

For a sense of the overall zeitgeist of the movement, read this piece.

It made a lot of sense on paper. There are so many smart people in the world and only a tiny fraction of them, the ones who happened to grow up in the bay area or who went to an elite school and studied CS, were able to engage in the creation of software. If someone could build a way to allow those nontechnical but smart people to build software, then it would unleash trillions of dollars of productivity into the world.

I was an early believer in the no-code movement. In 2020, I dropped out of school to become a founding engineer at Bubble, a company that would go on to become the strongest player in the "no-code" vertical. I got the job by cold-emailing the CEO. I ended up staying there through 2024. I watched us raise $100m and saw ARR skyrocket. We thought we were riding a rocketship.

Unfortunately, although people are still talking about this idea, the phrase that they're using to describe it is "vibe coding" rather than "no code". No-code isn't a fashionable term any more.

In this post, I'm gonna talk about why this happened.

Why is no code not fashionable any more?

Building an application requires engineering, if not coding.

Applications are complex

  • There's an irreducible amount of complexity inherent to any reasonably useful piece of software.
  • Types of complexity:
    • Product surface area (what happens when a user quits midway through onboarding after making a profile? How close together should we let patients schedule two appointments?)
    • Real-world messiness (we need to write a separate flow for the visa application for every country)
    • Architectural decisions (do we expect this data type to have a >100 read:write ratio?)
    • Design decisions (should this piece of UI disappear on page width <1080 to make room for the rest of the page?)
  • Good abstraction design can reduce complexity. But then we're forced to conclude that you can't write good applications without writing good abstractions; this implies that you do need to be technical in order to build software.
  • The best way to represent a complex piece of application logic is code, not a no-code app builder UI. It's easy to ctrl-f, and it makes it easy to repeat bits of logic.
  • The amount of irreducible complexity in an app is equal to the amount of context specific to that app.
  • Anything in an app that's not context-dependent is boilerplate, and can be implemented by the no-code framework.

Every no-code app requires custom code

  • Inevitably, any app built by no-code requires some bit of logic that isn't supported out of the box by a no-code framework. Some examples:
    • how many weekdays are there in between these two dates
    • read from the user's clipboard
    • trigger for when the app goes offline
  • Related: every no-code app inevitably requires an integration with a third-party vendor that isn't supported out-of-the box by no-code.
  • AI can mostly fix this problem.

There's a steep learning curve

  • Learning how to code is hard; learning how to no-code is easier but not easier. As we saw above, some complexity is inherent to software. Any software builder needs to learn concepts like relational databases, responsiveness, interfaces and APIs to have a reasonable shot at success. This takes a while.
  • The value prop of learning a no-code framework is made less appealing by the perception that the knowledge won't transfer off-platform, unlike coding.

Vibe-coding is a much better experience

  • AI can be a great engineer. It can take information about an application and make reasonable design decisions. It can write custom code for logic not covered by no-code frameworks. And it delivers results instantly.

Postmortem: what should we take away from no-code and what should we leave behind?

Keep

  1. Getting rid of code
    1. Getting rid of writing code was actually a great idea. It's just that we shouldn't have also gotten rid of interface/abstraction design, architectural decision making, and code as a source of truth.
  2. Abstracting away the true boilerplate + infrastructural complexity of building
    1. Lovable, etc have a long way to go before they automate hosting, keeping your code up-to-date, security, etc
  3. Visual WISYWIG editing
    1. Vibe coding hasn't figured this one out yet, but it's a much cleaner / more beautiful UX and necessary to design something actually world-class. Lovable needs to implement Figma-style WISYWIG.
    2. The highest-throughput communication medium of design is designs, not descriptions of designs.

Leave behind

  1. Providing a framework to do everything
    1. Implementations are free right now
  2. Proprietary languages and intermediate representations
    1. This is Bubble-specific
